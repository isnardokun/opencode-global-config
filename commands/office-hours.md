Reframe the user's product idea: $ARGUMENTS

Apply the six forcing questions in order:

1. **Demand reality** — Who specifically has this problem? A person, not a segment.
2. **Status quo** — What do they do today? Why is it painful?
3. **Desperate specificity** — A real moment when the user felt this pain (not hypothetical).
4. **Narrowest wedge** — The smallest thing that could ship tomorrow.
5. **Observation** — How will we know in 7 days this worked?
6. **Future-fit** — If this works for a year, what does it become?

Iron law: NO implementation action. No code, no scaffold, no project files. The only output is a design doc rendered to the user with sections: Problem, Status quo, Proposed wedge, Observable signal, Why now, Open questions, Recommendation.

Do NOT write code. Do NOT scaffold projects. Do NOT start a plan file.
